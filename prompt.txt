# TRAE Orbit：AI 随人走，算力随任务动

赛道：学习工作

## 1. 产品定位

**TRAE Orbit** 是一个跨设备 AI 任务调度系统。

它不是简单的远程桌面，也不是单一设备上的 AI 助手，而是让 AI 任务状态、上下文和执行进度跟随用户在手机、电脑、网页、云端之间流转。用户只需要提出目标、确认关键节点、验收结果；系统会根据任务复杂度、设备状态、隐私要求和可用算力，自动选择最合适的设备或云端环境执行任务。

一句话概括：

> AI 随人走，算力随任务动。

## 2. 想解决的问题

现在很多 AI 工具仍然和具体设备绑定。用户在电脑上开始的任务，切到手机后经常需要重新描述背景、重新上传文件、重新说明任务做到哪一步。

另一方面，不同任务需要不同资源。手机不适合处理大文件和长时间计算，云端不一定能访问本地文件、公司内网或浏览器登录态，电脑也不一定随时在线。用户现在往往需要自己判断任务该在哪台设备上处理，既麻烦，也容易中断。

TRAE Orbit 想解决的核心问题是：

> 让 AI 任务在多设备之间连续流转，并根据任务需求自动选择合适的算力执行。

## 3. 为什么想到做这个

最开始的想法是：能不能用手机远程操作电脑端 TRAE，让用户离开电脑后也能继续推进任务。

但进一步思考后发现，用户真正需要的不是在手机上远程看一块电脑屏幕，而是让 AI 工作现场跟随自己移动。任务需要轻量处理时，可以在手机或网页端完成；任务需要本地文件、浏览器登录态或电脑闲置算力时，就应该自动调度到电脑；任务需要更强算力时，也可以交给云端。

所以产品从“手机控制电脑”升级为：

> 跨设备 AI 任务连续性 + 智能算力调度。

## 4. 产品形态

TRAE Orbit 可以作为一套跨端系统存在：

| 端 | 作用 |
|---|---|
| 手机 App / 小程序 | 发起任务、查看进度、审批风险操作、接收结果 |
| Web 端 | 查看任务队列、管理设备资源池、继续编辑任务 |
| 电脑端客户端 | 读取本地文件、运行代码、调用浏览器、执行复杂任务 |
| 云端执行环境 | 处理大文件、长任务、高并发任务或需要更强算力的任务 |

产品不是把所有能力塞进一个端，而是让每个端做自己最适合的事。

## 5. 核心用户

TRAE Orbit 面向经常在多台设备之间切换的学习者、职场人、创作者和开发者。

典型用户包括：

| 用户 | 需求 |
|---|---|
| 学生 | 在电脑上整理资料，路上用手机继续复习和查看总结 |
| 职场人 | 电脑上处理文档和数据，外出后用手机查看进度 |
| 创作者 | 在不同设备间切换构思、整理素材、生成内容 |
| 开发者 | 在电脑上改代码，离开工位后用手机补充需求和审批任务 |
| 小团队成员 | 多人协作时，通过任务状态了解 AI 正在做什么 |

## 6. 使用场景

### 场景一：学习资料整理

用户在手机上说：

> 帮我整理这份课程资料，提炼重点、生成复习提纲。

系统判断资料较大，手机处理不合适，于是将任务调度到电脑或云端执行。用户在手机上查看进度，最后收到重点总结和复习提纲。

### 场景二：工作文档分析

用户在电脑上让 AI 分析一份表格，中途要离开办公室。任务状态同步到手机，用户在路上继续查看分析进度。如果需要确认输出格式，手机端弹出提示让用户选择。

### 场景三：开发任务接力

用户在电脑上开发项目，离开后在手机上补充一句：

> 顺便把移动端样式也检查一下。

TRAE Orbit 将补充需求加入原任务胶囊，电脑端继续执行。涉及安装依赖、删除文件或提交代码时，手机端提醒用户审批。

### 场景四：本地登录态任务

用户让 AI 打开某个已登录后台并截图分析。云端无法访问用户登录态，系统自动选择已登录的本地电脑执行，并将截图和结论同步回手机。

### 场景五：低电量与 Wi-Fi 情境

用户手机低电量，仍然发起一个复杂任务。系统检测到手机不适合执行，且用户电脑在线，于是自动将任务交给电脑处理。用户只在手机上接收摘要和关键提醒。

## 7. 核心功能

### 任务胶囊

任务胶囊是 TRAE Orbit 的核心概念。

每个 AI 任务都会被封装为一个独立的任务对象，里面记录任务目标、上下文、进度、权限、执行位置和结果。用户切换设备时，不是重新开始聊天，而是继续同一个任务胶囊。

任务胶囊包含：

| 内容 | 说明 |
|---|---|
| 任务目标 | 用户想完成什么 |
| 上下文 | 文件、对话、网页、代码路径、历史决策 |
| 当前状态 | 待执行、执行中、等待确认、已完成、失败 |
| 执行位置 | 手机、本地电脑、云端、服务器 |
| 权限需求 | 是否需要读写文件、执行命令、访问浏览器 |
| 中间结果 | 日志、摘要、截图、文件变更 |
| 风险等级 | 低风险、中风险、高风险 |
| 下一步动作 | 继续、等待用户、切换设备、重试、结束 |

### 全局任务状态协议

TRAE Orbit 需要一套全局任务状态协议，让任务可以跨设备识别、同步和恢复。

它解决的是：

> 任务不绑定设备，而是绑定用户目标。

任务状态可以在手机、电脑、网页端之间同步。用户在哪个设备上打开，都能看到任务做到哪一步、下一步需要什么、当前由哪台设备执行。

### 智能算力调度

系统根据任务需求自动选择执行位置。

| 任务类型 | 推荐执行位置 |
|---|---|
| 简单问答、摘要、提醒 | 手机端或轻量云端 |
| 读取本地文件、分析本地项目 | 用户电脑 |
| 代码修改、运行测试 | 本地电脑或远程开发环境 |
| 大文件分析、批量处理 | 云端或高性能电脑 |
| 需要浏览器登录态的网站操作 | 已登录的本地设备 |
| 高风险操作 | 授权设备执行 + 手机审批 |

用户不需要自己判断“这个任务该在哪跑”，系统会自动推荐，并解释原因。

### 设备资源池

用户的设备可以组成一个个人 AI 资源池。

资源池中的设备可以包括：

| 设备 | 能力 |
|---|---|
| 手机 | 随身入口、审批、通知、传感器 |
| 办公电脑 | 本地文件、公司内网、浏览器登录态 |
| 家用电脑 | 闲置算力、个人文件、长任务 |
| 云端环境 | 高算力、大文件、长时间任务 |
| NAS / 服务器 | 存储资料、离线任务、备份 |

TRAE Orbit 不依赖单一品牌协议，而是基于账号和网络连接。只要设备登录同一账号，并能通过局域网、公网或中继网络连接，就可以加入资源池。

### 可解释调度

系统每次调度任务时，都应给用户一个简单解释。

例如：

> 本任务包含大文件分析，手机处理较慢；检测到云端算力可用，已建议转到云端执行。

或者：

> 本任务需要访问已登录后台，云端无法使用你的浏览器登录态，已调度到本地电脑执行。

这样可以提升用户信任感，避免 AI 调度变成黑盒。

### 风险审批

任务执行时，根据风险等级决定是否需要用户确认。

| 风险等级 | 操作 | 处理方式 |
|---|---|---|
| 低风险 | 总结、搜索、生成草稿 | 自动执行 |
| 中风险 | 修改文件、运行测试、安装依赖 | 首次确认或按规则确认 |
| 高风险 | 删除文件、提交代码、上传数据 | 每次确认 |
| 极高风险 | 批量删除、未知脚本、敏感数据外传 | 默认禁止或强制二次确认 |

手机端是最适合做审批的设备。用户不用远程操作电脑，只需要确认关键动作。

### 情境感知主动服务

这是未来扩展能力。

在用户授权的前提下，系统可以结合手机传感器和环境信号提供主动服务。

| 信号 | 可以判断什么 | 服务方式 |
|---|---|---|
| Wi-Fi | 是否到家、到公司、进入可信网络 | 到公司后恢复工作任务 |
| 时间 | 当前是否适合打扰 | 深夜只推送低打扰摘要 |
| 位置 | 是否在学校、办公室、通勤路上 | 通勤时推送适合手机查看的任务 |
| 蓝牙 | 是否连接耳机、车机、键盘 | 连接耳机时语音播报进度 |
| 电量 | 手机是否适合执行任务 | 低电量时转给电脑或云端 |
| 网络状态 | 是否适合上传大文件 | Wi-Fi 下自动同步大文件 |

这让 TRAE Orbit 不只是被动等待指令，而是能根据用户状态提供更合适的提醒和执行方式。

## 8. 产品流程

一个完整任务流程如下：

1. 用户在任意设备上发起任务。
2. 系统创建任务胶囊。
3. 系统记录任务目标、上下文、权限需求和初始状态。
4. 调度器判断任务类型、复杂度、隐私要求和可用设备。
5. 系统选择最合适的执行端。
6. 执行端开始处理任务。
7. 手机端或当前设备同步进度。
8. 涉及风险操作时，用户收到审批提醒。
9. 任务完成后，结果同步回用户当前设备。
10. 用户可以继续追问、接管任务或结束任务。

## 9. 差异化

TRAE Orbit 的关键差异是：

> 它迁移的是任务，不是屏幕。

| 常见方案 | 局限 | TRAE Orbit 的不同 |
|---|---|---|
| 远程桌面 | 把电脑屏幕传到手机，操作负担仍在用户身上 | 迁移任务状态，用户只做目标输入、审批和验收 |
| 普通 AI 聊天 | 会话容易割裂，换设备后上下文不连续 | 任务胶囊记录上下文、状态和下一步 |
| 云端 AI Agent | 云端有算力，但不一定能访问本地文件和登录态 | 可在手机、电脑、云端之间动态调度 |
| 品牌生态互联 | 依赖同品牌设备和专用系统协议 | 基于账号和网络连接，跨品牌、跨系统 |
| 普通提醒工具 | 只能按时间提醒 | 结合任务状态和传感器信号，提供主动服务 |

## 10. 价值与意义

### 效率提升

TRAE Orbit 可以减少设备切换带来的重复沟通、重复上传和重复操作。用户在电脑上开始的任务，可以在手机上继续查看；手机上发起的任务，也可以自动交给电脑或云端执行。

### 降低门槛

用户不需要理解复杂的设备配置、云端环境或远程连接方式。系统会根据任务自动选择执行位置，并给出调度解释。用户只需要描述目标和确认关键操作。

### 更适合真实工作

真实的学习和工作并不是只发生在一台设备上。人会移动，设备会切换，任务会中断，环境会变化。TRAE Orbit 的价值就是让 AI 适应这种真实状态，而不是要求用户一直坐在某台电脑前。

### 更安全可控

通过任务胶囊、权限记录、风险等级和手机审批，系统可以在自动执行和用户控制之间取得平衡。敏感任务优先本地执行，高风险操作必须确认，让用户知道 AI 正在做什么、为什么这么做。

## 11. MVP 版本

第一版可以先做一个轻量 MVP：

| 模块 | 功能 |
|---|---|
| 任务入口 | 手机或网页端发起任务 |
| 任务胶囊 | 记录目标、上下文、状态和下一步 |
| 设备列表 | 展示手机、电脑、云端是否在线 |
| 调度规则 | 根据任务类型选择执行端 |
| 进度同步 | 手机端查看任务状态 |
| 风险审批 | 高风险操作需要用户确认 |
| 结果回传 | 任务完成后生成摘要和结果卡片 |

MVP 不需要一开始实现复杂的传感器主动服务，也不需要完整跨品牌底层协议。可以先验证核心问题：

> 用户是否愿意把任务交给系统调度，而不是自己决定在哪台设备上处理？

## 12. 一句话总结

> TRAE Orbit 是一个跨设备 AI 任务调度系统，让 AI 随人走，算力随任务动。它把每个任务封装成可流转的任务胶囊，在手机、电脑和云端之间智能接力，让用户在任意设备上都能继续同一个 AI 工作现场。
